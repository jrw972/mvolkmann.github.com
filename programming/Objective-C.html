Objective-C

Things I Hate

  1. having a .h and .m file for each class violates DRY
  2. allows all C syntax including pointer arithmetic
  3. asterisk required in declaration of all object variables, but -> is not used to access their methods
  4. literal string syntax @"text"
  5. @property and @synthesize should be combined
  6. no automatic garbage collection under iOS
  7. very verbose methods in provided classes like NSString
  8. instance variables are defined in interfaces, not classes
  9. syntax difference between using categories and protocols is arbitrary and not easy to remember; <protocol> vs. (category)
  10. method parameter and return types inside parens
  11. method type designator conflicts with UML syntax; + means class method, not public; - means instance method, not private
  12. the word "mutable" is used incorrectly in collection class names; means size can change; not being mutable means size is fixed; data can change regardless
  13. categories seem like they should be used to add a set of methods to multiple classes, but it appears they are usually used to add methods to a single class
  14. standard categories are needed for builtin classes like NSString to provide less verbose methods; it seems many people roll their own
  15. the syntax for "block closures" is so ugly that Objective-C may as well not have closures at all
  16. must import .h files into corresponding .m files
  17. boolean values are YES and NO, not true and false
  18. no support for namespaces or packages so class naming conventions are used instead
  19. can't use [ ] syntax for all calls; some require C syntax so code will contain a mixture of calling syntaxes
  20. when a method calls another method in the same class, that method definition must appear earlier in the source file or warnings will appear (but it will still work)
  21. no simple way to concatenate strings
  22. no support for generics
  23. no private methods
  24. implementing mutually referential classes requires tricks
